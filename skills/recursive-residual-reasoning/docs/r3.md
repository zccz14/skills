# Recursive Residual Reasoning

Recursive Residual Reasoning, abbreviated as R3, is a reasoning framework for systems whose current explanation space is incomplete.

Its core idea is:

> In any reasoning layer, unresolved material must stay inside an explicitly defined reasoning space. If the current layer cannot explain it, the unexplained part must be marked as a residual and expanded into the next reasoning layer.

In short:

> Unexplained does not mean ignored. It means the current reasoning space is incomplete.

R3 does not assume a system starts with enough explanatory power. It requires every unresolved part to remain traceable and expandable.

## 1. Algebraic Model

R3 can be modeled as:

```text
R3 = (U, Ω, X, τ, T)
```

Where:

| Symbol | Meaning |
| --- | --- |
| `U` | Universe of reasoning objects: observations, hypotheses, evidence, nodes, conclusions, actions, stop reasons, and traces. |
| `Ω` | Closed operations over `U`, such as classify, explain, decompose, combine, test, expand, close, and trace. |
| `X` | Residual elements: explicit parts not explained by the current reasoning space. |
| `τ` | Residual expansion operator: maps a residual into a new reasoning space. |
| `T` | Terminal set: conclusions, actions, formal closures, or bounded stop reasons. |

The essential operation is:

```text
unknown → residual → next reasoning space
```

## 2. Core Principles

An R3 system should satisfy these principles.

### Layer Closure

At any layer `L`, the unresolved target must be routed to an explicit node, a combination space, or a residual:

```text
Target_L ∈ { C1, C2, ..., Cn, Cmb_L, X_L }
```

Where:

| Symbol | Meaning |
| --- | --- |
| `C1 ... Cn` | Atomic nodes in the current layer. |
| `Cmb_L` | Combination space for interactions between known nodes. |
| `X_L` | Residual at the current layer. |

### Residual Explicitness

When current nodes cannot explain the target, the system must not skip, blur, or hide the unresolved part. It must name an explicit residual `X_L`.

### Residual Recursion

`X_L` is not a default terminal state. It becomes the object of the next reasoning layer:

```text
X_L → Space_(L+1)
```

The next layer must define its own nodes, combination space, residual, observables, rules, actions, and stop criteria.

### Finite Closure

The recursion should terminate in finite depth at one of these states:

- a conclusion
- an action
- a formal closure
- a modifiable rule, threshold, state, relation, or representation
- a bounded stop reason

## 3. Node Contract

Every non-residual node should bind at least three fields:

```text
C_i → (O_i, R_i, A_i)
```

Where:

| Field | Meaning |
| --- | --- |
| `O_i` | Observable: how the issue can be seen. |
| `R_i` | Rule: how the issue is judged. |
| `A_i` | Action: what changes after the issue is confirmed. |

A valid node does not merely say that something might explain a problem. It states:

1. how to observe the issue
2. how to judge it
3. how to act after confirmation

## 4. Combination Space

The combination space `Cmb_L` handles cases where known nodes do not individually explain a phenomenon, but their combination does.

Common combination subspaces are:

```text
Cmb_L → { T_L, W_L, I_L, S_L }
```

Where:

| Symbol | Meaning |
| --- | --- |
| `T_L` | Temporal or ordering issue. |
| `W_L` | Weight, priority, or allocation issue. |
| `I_L` | Interaction issue. |
| `S_L` | State, mode, or context transition issue. |

`Cmb_L` must not become a new trash bucket. If a target is routed to a combination space, the system must further decompose it into a specific relation, such as ordering, priority, interaction, state transition, feedback, shared context, or another explicit modifiable combination rule.

## 5. Emergent Behavior

R3 treats emergent behavior as a first-class combination case.

If:

```text
P(A) = not observed
P(B) = not observed
P(A, B) = observed
```

Then the explanation should not be forced into `A` or `B` alone. It should be routed to an interaction residual or combination node:

```text
emergent phenomenon
→ combination space
→ interaction, ordering, feedback, shared state, threshold, or context dependency
→ next reasoning space
```

## 6. Multi-Node Change Rule

R3 distinguishes a single-node explanation from a multi-node intervention.

If an action changes, tests, or depends on multiple nodes at the same time, the trace must not claim that it validates one node alone. It must route through a combination space unless the effects are explicitly separated by evidence and trace records.

Use this rule:

```text
single handled node + unchanged neighbors → single-node trace
multiple handled nodes or inseparable effects → combination trace
```

Every non-trivial trace should record:

- handled node
- not-handled neighboring nodes
- changed nodes
- reason the trace is single-node or combination-space

This prevents a reasoning system from modifying several parts of its explanation space and then assigning the result to only one of them.

## 7. Evidence Availability Boundary

R3 separates explanatory evidence from decision-available evidence.

Evidence can be used for retrospective explanation, but it must not be silently treated as evidence that was available at the time of a decision, intervention, or real-time judgment.

Every important evidence item should be labeled with availability:

| Availability | Meaning |
| --- | --- |
| `before_decision` | Available before the relevant decision or action. |
| `during_decision` | Available at the moment of decision or action. |
| `after_decision` | Known only after the decision or action. |
| `synthetic_or_inferred` | Produced by later modeling, inference, simulation, or reconstruction. |

Use this rule:

```text
diagnostic evidence may explain a trace
operational evidence may guide an action
post-hoc evidence cannot be treated as operational evidence unless availability is proven
```

This rule is domain-neutral. It prevents an explanation from using information that did not exist inside the original decision boundary.

## 8. Mandatory Trace Record

Every non-trivial R3 analysis must produce a trace record. A durable conclusion, frozen decision, accepted explanation, or project-level claim is incomplete without the full trace.

The minimum trace fields are:

| Field | Meaning |
| --- | --- |
| `r3_node` | Node hit by the current reasoning step. |
| `parent_node` | Parent node or reasoning space. |
| `handled_node` | Node this trace actively explains, tests, or changes. |
| `not_handled_nodes` | Neighboring nodes explicitly left unchanged or unclaimed. |
| `changed_nodes` | Nodes changed by the action or intervention. |
| `observable` | Observable used for the step. |
| `rule` | Judgment rule used for the step. |
| `action` | Action, conclusion, or next move. |
| `evidence` | Evidence reference. |
| `evidence_availability` | When the evidence became available relative to the relevant decision or action. |
| `evidence_metric` | Key measurement, score, observation, or citation, when applicable. |
| `residual` | Named unresolved part, if any. |
| `expanded_to` | Next reasoning space generated from the residual. |
| `next_if_failed` | Next node, residual, or combination space if the current trace fails. |
| `closed` | Whether the trace has reached a terminal state. |
| `closure_type` | `conclusion`, `action`, `formal_closure`, or a bounded stop reason. |

## 9. Residual Expansion Rules

When a target is routed to `X_L`, the system must:

1. treat `X_L` as the target of the next reasoning space
2. define a new node set
3. define a new combination space
4. define a new residual
5. bind observables, rules, and actions to non-residual nodes
6. define stop reasons
7. continue the trace

Formally:

```text
X_L → Space_(L+1)

Target_(L+1) ∈ { C1^(L+1), C2^(L+1), ..., Cm^(L+1), Cmb_(L+1), X_(L+1) }
```

## 10. Stop Reasons

If a residual cannot be expanded, the system must not end with vague unknown. It must use a bounded stop reason.

Common stop reasons:

| Stop reason | Meaning | Next move |
| --- | --- | --- |
| `evidence_lack` | Required observations or records are missing. | Add collection, instrumentation, or research. |
| `model_lack` | The system lacks a useful abstraction. | Introduce a new model, concept, or representation. |
| `sample_lack` | Available cases are not enough to judge. | Gather more cases or wait for more observations. |
| `cost_limit` | Further recursion costs more than it is worth now. | Pause recursion and record risk. |
| `boundary_limit` | The issue is outside the defined system boundary. | Expand the boundary or hand off to another system. |

## 11. Decomposability

Complex observables should be decomposable into simpler propositions:

```text
P = f(p1, p2, ..., pk)
```

The combination function `f` may be a boolean rule, threshold, statistical judgment, score, confidence model, or another explicit rule.

## 12. Low Overlap

Observables in the same layer should have low explanatory overlap. If multiple nodes can explain the same target, the system must define at least one of:

- priority rule
- joint judgment rule
- conflict resolution rule
- multi-cause attribution weight
- route to a combination space

R3 does not require perfect orthogonality. It requires the routing rules to be judgeable, sortable, and executable.

## 13. Relative Completeness

R3 completeness is relative, not absolute.

A system is relatively complete when, under a defined boundary, observation capacity, node library, maximum recursion depth, and reasoning cost limit, each target class can terminate in finite depth at a non-residual node, a combination rule, or a bounded stop reason.

Relative completeness depends on:

| Condition | Meaning |
| --- | --- |
| boundary | What the system is responsible for explaining. |
| observation capacity | What the system can observe. |
| node library | Which nodes are currently defined. |
| maximum depth | How far recursion may go. |
| reasoning cost | How much effort is worth spending. |
| target classes | Which issue classes the system promises to cover. |

An operational R3 system should maintain coverage records for target classes:

| Field | Meaning |
| --- | --- |
| `target_class` | Class of observation or problem. |
| `initial_route` | First node, combination space, or residual. |
| `terminal_path_exists` | Whether a finite path to a terminal state is defined. |
| `residual_if_not` | Residual used when no terminal path exists. |
| `last_reviewed` | Last time the coverage claim was checked. |

## 14. Persistent System State

R3 is not only a single-analysis procedure. In project settings, it should maintain persistent reasoning state.

The persistent state should include:

- current node library
- current residual library
- combination spaces and relation subspaces
- known traces and accepted conclusions
- open residual backlog
- priority expansions
- coverage records
- deprecated nodes and replacement relations

Stable node IDs should not be changed after traces reference them. If a node becomes obsolete, mark it deprecated and link it to its replacement instead of renaming it in place.

## 15. Minimal Implementation Checklist

To implement R3 in a project, define:

1. system boundary
2. target classes
3. observation capacity
4. maximum recursion depth
5. reasoning cost limit
6. first-layer node set
7. combination space
8. residual nodes
9. observables for non-residual nodes
10. judgment rules for observables
11. actions for confirmed nodes
12. residual expansion rules
13. stop reasons
14. trace output format
15. conflict resolution policy
16. project-level R3 data file
17. evidence availability labels
18. handled and not-handled node policy
19. multi-node change routing policy
20. known trace and open residual backlog
21. target coverage audit
22. node deprecation policy

## 16. Complete Definition

Recursive Residual Reasoning is a layered reasoning framework in which:

1. each layer routes the target to explicit nodes, a combination space, or a residual
2. each non-residual node binds observables, rules, and actions
3. combination spaces must decompose into explicit relation problems
4. residuals must expand into new reasoning spaces or bounded stop reasons
5. recursion must terminate in finite depth under declared boundary and cost limits
6. the trace from observation to closure is part of the result
7. multi-node changes must route through combination space unless effects are separated
8. evidence availability must be recorded when a trace informs a decision or action
9. persistent projects must maintain R3 node data, relation data, known traces, and open residuals

## 17. Short Form

R3 does not allow a system to end at "I do not know".

Every unknown must become a named residual, and every residual must either generate the next reasoning space or close with an explicit bounded stop reason.
